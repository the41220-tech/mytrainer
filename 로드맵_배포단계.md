# 배포 로드맵 — 개인용 → 다중 사용자

> 결정(2026-06): **지금은 개인용으로만 배포**(데이터 영속 저장만 확보), **나중에 여러 명도 가능하게** 전환.
> 핵심: 아키텍처는 이미 다중 사용자까지 열려 있어, 단계 간 변경은 대부분 **인증·영속성 "설정"** 뿐.

## 0. 데이터는 어디에 저장/로드되나 (요약)
- **별도 DB 서비스 없음. SQLite 파일이 곧 무료 임베디드 DB** (서버·가입·구독 0원).
- **식별**: 요청 헤더의 Key/Token → **저장**: `<MYTRAINER_DB_DIR>/users/<키의 sha256 앞16자>.db` → **로드**: 같은 키 = 같은 파일. (키 원문은 저장 안 함)
- 키가 없으면 → **단일 파일** `MYTRAINER_DB_PATH` (없으면 `~/.mytrainer/mytrainer.db`). ← **개인용은 이 경로 1개.**
- 연결은 파일경로별 LRU 캐시(기본 200, `MYTRAINER_MAX_CACHE`).
- ⚠️ 클라우드 디스크는 보통 휘발성 → **영속 볼륨을 붙여야 데이터가 남는다**(감사 C4).

## 1. 단계

### Phase 0 — 개인용 (지금 배포) ✅ 목표
- **인증**: 없음 (`MYTRAINER_REQUIRE_KEY` 미설정).
- **DB**: 단일 SQLite 파일. **`MYTRAINER_DB_PATH`를 영속 볼륨 경로로 고정**.
- **데이터**: 나 하나. 격리 신경 쓸 것 없음.
- **목적**: 직접 쓰며 버그를 잡는다(iterate).
- **실행**: `MYTRAINER_DB_PATH=/data/mytrainer.db PORT=8080 node dist/http.js`
- **이미 됨**: 22 tool(기록·ML·분석·경고·동기부여), v2 예측 모델, 데모 시드.
- **남은 일**: 클라우드에 **영속 볼륨 마운트** + 배포. (코드 변경 0)

### Phase 1 — 다중 사용자 (토큰별 격리, 설정만 켜기)
- **이미 구현됨**: store가 토큰→사용자별 `.db`로 격리, LRU 캐시, `MYTRAINER_REQUIRE_KEY` 가드.
- **켜기**: `MYTRAINER_REQUIRE_KEY=true` + 헤더 이름을 PlayMCP Key/Token '커스텀 헤더'와 일치(`x-api-key`). `MYTRAINER_DB_DIR`를 영속 볼륨으로.
- **전제**: **사용자마다 토큰이 달라야** 진짜 분리됨. PlayMCP Key/Token이 **공용 1개**면 모두 같은 파일 공유 → 분리 안 됨 → Phase 2 필요.
- **코드 변경**: 0 (설정만).

### Phase 2 — 진짜 멀티유저 (OAuth 정체성)
- **인증**: **OAuth2 PKCE** — 사용자별 로그인 → 사람마다 고유 식별자 → 완전 격리.
- **영속성**: 영속 볼륨, 또는 **외부 DB** — **Turso**(클라우드 SQLite, 무료티어) / Supabase·Neon(Postgres 무료티어).
- **코드 변경**: `http.ts`에 OAuth 핸들러 추가, (외부 DB 채택 시) `store` 어댑터 1개. — `Store` 인터페이스 추상화 덕에 엔진·tool은 그대로.
- **시점**: 본선 진출 또는 공개 다중 사용자 시.

## 2. 전환 비용 한눈에
| 단계 | 인증 | DB | 코드 변경 | 시점 |
|---|---|---|---|---|
| **Phase 0 개인** | 없음 | 단일 SQLite(영속) | **0** | **지금** |
| Phase 1 다중(토큰별) | Key/Token | 사용자별 SQLite(영속) | 0 (설정만) | 다중 필요 시 |
| Phase 2 멀티유저 | OAuth PKCE | 영속볼륨/Turso/PG | OAuth + (선택)어댑터 | 본선/공개 |

**요지: 아키텍처는 이미 Phase 2까지 열려 있다**(store 추상화 · 토큰 격리 구현 · stateless HTTP). 지금은 **Phase 0만 켠다** — 영속 저장만 확보하면 끝.
